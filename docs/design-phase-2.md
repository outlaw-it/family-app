# Family Brain — Phase 2 design

> Status: **draft for review** (2026-06-23). This is a design, not a build. It turns
> the current recall-only demo into something a real household uses daily: the
> assistant can **do** things (MCP tools), **remember** smarter (semantic + auto-capture),
> and **feel** like a product (streaming + conversation history).

## 1. What we're adding (and what we're deliberately not)

Chosen for this phase:

1. **MCP tool use** — the assistant can call linked MCP servers (M365, web search,
   smart home, anything custom) mid-conversation, not just recall stored notes.
2. **Smarter memory** — semantic recall (embeddings) + automatic memory capture, with
   keyword recall kept as a zero-config fallback.
3. **Streaming + conversation history** — token-by-token responses and a sidebar to
   browse/resume past chats (the data is already stored, just not surfaced).

Explicitly deferred but **designed around**:

- **Real auth** — you didn't prioritise it, but "one real household + real email +
  actions" needs *some* gate. See §6. The design makes it a small, optional bolt-on,
  not a rewrite.
- **Multi-household / SaaS** — out of scope. Single household stays the unit.

## 2. Guiding constraints (don't break these)

- **THE PRIVACY RULE stays the one chokepoint.** Every new read — vector recall,
  conversation history, tool results that touch memory — routes through
  `allowedSpaceIds()` in `src/lib/memories.ts`. No feature gets its own `SELECT`.
- **Cloudflare Workers reality:** no local processes. That means **remote MCP servers
  only** (Streamable HTTP / SSE transport) — no stdio child processes. This shapes the
  whole MCP design (§3).
- **Zero-config ethos survives.** "Clone, set `DATABASE_URL` + a model key, it works"
  must still be true. Every new capability **degrades gracefully** when its key/most
  isn't configured: no embedding key → keyword recall; no MCP servers linked → today's
  behaviour; no household secret → demo switcher.
- **Provider abstraction stays small.** Adding a provider should still be "one entry +
  one case". Tool-calling and streaming are added as *capabilities of the existing
  providers*, not a second framework.

## 3. MCP tool use

### 3.1 My recommendation on direction (you asked me to advise)

**Build the client direction first; add the server direction later, only if you want it.**

- **App as MCP _client_** (assistant connects OUT to tools) is what "link MCP" almost
  always means and is the whole point of "let it *do* things". This is the priority.
- **App as MCP _server_** (expose family memories to Claude Desktop etc.) is a nice
  bonus but it's a *different audience* (you, in another AI app) and it re-exposes the
  privacy rule across a network boundary — more risk, less daily value. Park it behind
  a `Phase 2.5` flag. The memory layer is already clean enough to wrap later.

So: **client now, server maybe-later.** The rest of §3 is the client.

### 3.2 Transport: remote servers only

Because the app runs on Workers, it talks to MCP servers over **Streamable HTTP/SSE**,
never stdio. Practical consequence: each integration must be reachable as a URL.

| Integration | How it shows up | Auth | Lift |
|---|---|---|---|
| Web search | Hosted MCP endpoint (or a thin built-in tool) | API key header | **Low** — do first |
| Custom / smart home | "Add MCP server by URL" | API key header / none | **Low** |
| Microsoft 365 | Remote MCP server (Graph) | **OAuth 2.0** per user | **High** — do last |

This is why M365 is sequenced last: it's not "another tool", it's the one that drags in
OAuth + a per-member token store (§6 ties into this).

### 3.3 Data model

```sql
CREATE TABLE mcp_servers (
  id            INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  scope         TEXT NOT NULL CHECK (scope IN ('family','personal')),
  owner_member_id INTEGER REFERENCES members(id) ON DELETE CASCADE,  -- null for family
  name          TEXT NOT NULL,
  url           TEXT NOT NULL,
  transport     TEXT NOT NULL DEFAULT 'http',     -- 'http' (streamable) | 'sse'
  auth_kind     TEXT NOT NULL DEFAULT 'none',      -- 'none' | 'api_key' | 'oauth'
  enabled       BOOLEAN NOT NULL DEFAULT true,
  created_at    TEXT NOT NULL DEFAULT to_char(now(),'YYYY-MM-DD HH24:MI:SS')
);
-- Secrets (api keys, oauth tokens) NEVER go in this table. They live in a separate
-- token store keyed by (server_id, member_id), or in env for shared/family servers.
```

A server linked at `family` scope is usable by everyone; at `personal` scope only by its
owner — **same privacy model as memories**, reused.

### 3.4 The agent loop (the real architectural change)

Today `chat()` returns a `string` in one shot. Tool use needs a **loop**: the model can
ask to call a tool, we run it, feed the result back, and repeat until it produces final
text. New module `src/lib/agent.ts` owns this; `providers.ts` gains a tools-aware,
streaming call but stays per-provider-simple.

```
runAgent({ member, persona, scope, messages, tools }) :  async generator of events
  1. assemble system prompt (persona + recalled memories, as today)
  2. gather tools = built-ins + enabled MCP servers for (member, scope)
  3. loop:
       stream a model turn WITH tools
       ├─ text delta            → yield {type:'text', delta}
       ├─ model calls tool(s)   → yield {type:'tool_call', …}
       │                          execute via MCP client (timeout + error capture)
       │                          yield {type:'tool_result', …}; append; continue loop
       └─ model stops with text → yield {type:'done'}; break
```

Normalised internal shapes (provider adapters map to/from these):

```ts
interface Tool { name: string; description: string; inputSchema: JSONSchema }
interface ToolCall { id: string; name: string; input: unknown }
type AgentEvent =
  | { type: 'recall'; memories: Memory[] }        // emitted first — drives the panel
  | { type: 'text'; delta: string }
  | { type: 'tool_call'; call: ToolCall }
  | { type: 'tool_result'; id: string; ok: boolean; preview: string }
  | { type: 'saved_memory'; memory: Memory }
  | { type: 'done'; conversationId: number }
```

Both Anthropic (`tools` + `tool_use`/`tool_result` blocks) and OpenAI/Azure
(`tools` + `tool_calls`) support this; the adapter is where their shapes differ. Local
models that don't support tools simply get an empty tool list — graceful.

### 3.5 Safety: tools can act on the real world

- **Read vs write tools.** Classify tool calls; **writes/actions require confirmation**
  by default (a "Allow this?" step in the UI) unless the user marks a server trusted.
  Critical once email-send / smart-home are in play.
- **Tool results are transparent**, exactly like recall: the UI shows *what tool ran and
  what it returned*, mirroring today's "which memory was used" feature. This is the
  product's signature — extend it to tools.
- **Per-tool timeout + error isolation**: a slow/broken MCP server yields a
  `tool_result{ok:false}` the model can recover from, never hangs the chat.

## 4. Smarter memory

### 4.1 Semantic recall (pgvector)

Supabase ships **pgvector**. Add a nullable embedding column; embed on write; search by
similarity — **inside the same privacy-scoped query** so the rule holds.

```sql
CREATE EXTENSION IF NOT EXISTS vector;
ALTER TABLE memories ADD COLUMN embedding vector(1536);   -- nullable
-- recall (privacy preserved by the space_id filter, as today):
SELECT m.* FROM memories m
WHERE m.space_id = ANY(${allowedAndScoped})
ORDER BY m.embedding <=> ${queryEmbedding}        -- cosine distance
LIMIT 5;
```

- **Provider:** OpenAI `text-embedding-3-small` (cheap, 1536-d) is the default;
  pluggable like `chat()`. Embeddings are computed server-side only.
- **Graceful fallback:** no embedding key, or memory has no embedding yet → fall back to
  today's keyword `searchMemories`. Both live behind one `recall()` function so the chat
  route doesn't care which ran. **Zero-config still works.**
- **Backfill:** a tiny "re-embed all" maintenance call for when you first add a key.

### 4.2 Automatic capture (with review, not silent)

After an assistant turn, a cheap extraction pass proposes candidate memories
("Leo's new soccer coach is Sam, Tuesdays 4pm"). Rather than silently storing (which
pollutes memory and breaks transparency), surface them as **suggested** memories the
user accepts/edits/dismisses.

```sql
ALTER TABLE memories ADD COLUMN status TEXT NOT NULL DEFAULT 'active';  -- 'active' | 'suggested'
ALTER TABLE memories ADD COLUMN source TEXT NOT NULL DEFAULT 'manual';  -- 'manual' | 'chat' | 'auto'
```

Suggested memories don't show in recall until accepted. A setting can flip to
"auto-accept" for households that want it hands-off. Keeps the "explicit `remember
that…`" path working unchanged.

## 5. Streaming + conversation history

### 5.1 Streaming

`/api/chat` becomes a **Server-Sent Events** stream emitting the `AgentEvent`s from §3.4.
Order matters for UX: emit `recall` first (panel lights up instantly), then `text`
deltas, then any `tool_*`, then `saved_memory` + `done`. The `Chat` component consumes
the stream; everything it shows today (reply, used-memories, saved-memory) is just
events now. Works on Workers (SSE is supported under `nodejs_compat`).

### 5.2 Conversation history (data already exists)

`conversations` + `messages` are already written — we just expose them:

- `GET /api/conversations` — current member's conversations (scoped by `member_id`;
  privacy is per-member already). Title = first user message, truncated.
- `GET /api/conversations/[id]` — its messages, 404 if not the member's (same
  no-leak rule as memories).
- New **left sidebar**: list, click to resume (loads history, continues the same
  `conversationId` the chat route already accepts), "New chat" button.

Layout becomes: `[Conversations] | [Chat] | [Memory panel]`.

## 6. Identity — the flagged prerequisite (your call)

You deselected real auth but chose real daily use + real M365. Recommendation: a
**minimal optional gate**, not a full auth system.

- **If `HOUSEHOLD_SECRET` is unset** → today's behaviour exactly (demo switcher, zero
  setup). Nothing changes; the demo still clones-and-runs.
- **If set** → a simple unlock: pick your member + enter a per-member PIN (hash stored
  on `members`). Sets a signed session cookie that replaces the spoofable `x-member-id`
  header. ~Half a day's work, no external dependency.
- **Required before** enabling any **write/action** tool or **M365** (which carries real
  email). Read-only tools (web search) are fine without it.

This respects "keep it simple" while making "reads your real email" not a free-for-all.
If you'd rather go full **Supabase Auth** (magic links, fits the stack) we can — it's
more robust but breaks pure zero-config. **Decision needed.**

## 7. Provider abstraction — how it evolves

Keep the shape; add two capabilities:

- `streamChat(req, { tools })` → async iterator of normalised events. `chat()` stays as
  a thin "collect the stream into a string" wrapper so nothing that uses it breaks.
- Each provider adapter maps its native tool/stream shape to the normalised events.
- Local models: advertise no tools, no streaming-of-tools — degrade to plain streaming.

Adding a provider is still "one `PROVIDERS` entry + one `case`", now implementing
`streamChat` instead of `chat`.

## 8. Data model — all changes in one place

```
mcp_servers (new)                              -- §3.3
memories.embedding   vector(1536) NULL          -- §4.1
memories.status      'active'|'suggested'       -- §4.2
memories.source      'manual'|'chat'|'auto'     -- §4.2
members.pin_hash     TEXT NULL                   -- §6 (only used if HOUSEHOLD_SECRET set)
messages.tool_calls  JSONB NULL                  -- §3.4 transparency/history of tool runs
+ CREATE EXTENSION vector
```

All additive + nullable → `migrate()` stays idempotent, existing rows unaffected, seed
unchanged. No destructive migration.

## 9. Suggested sequence (each step is a usable slice)

1. **Streaming + history** — ✅ **BUILT (2026-06-23).** SSE streaming in `/api/chat`
   (`streamChat()` in `providers.ts`); `chat()` kept as a collect-the-stream wrapper.
   New `conversations.ts` + `/api/conversations[/[id]]` (member-scoped, no-leak).
   `ConversationList` sidebar + resume wired through `page.tsx`. tsc + `next build`
   clean. Not yet smoke-tested against a live DB.
2. **Semantic recall** — pgvector column + `recall()` with keyword fallback. Invisible
   if no embedding key; better answers if there is one.
3. **MCP client core** — `agent.ts` loop + `mcp_servers` + Settings UI to link a server +
   transparent tool display. Prove it with **web search** (read-only, API-key, low risk).
4. **Auto-capture (suggested memories)** — small, leans on §2 work.
5. **Optional gate (§6)** — before the next step.
6. **M365 over OAuth** — the heavy one; do it last, on top of a proven MCP loop.
7. *(Optional, later)* expose Family Brain **as** an MCP server.
